# **从 JVM 分代回收到 Agent 长期记忆：一种更接近人类记忆的设计思路**

最近在思考 AI Agent 的长期记忆机制时，我一直有一个感觉：现在很多 Agent 记忆系统还停留在“把东西存下来”的阶段，但真正的问题其实不是“存不存”，而是“什么东西应该被留下来，什么东西应该被忘掉，什么东西应该被整理成技能”。

如果一个 Agent 只是把所有聊天记录都保存下来，它当然不会真的丢失信息，但它也不一定会变聪明。因为信息太多以后，真正有用的东西反而会被淹没。相反，如果 Agent 只保留几条用户偏好，又会显得非常健忘，明明一起做过很多事情，但下次还是像第一次见面一样。这两种情况都不理想。

我觉得一个更合理的长期记忆系统，应该不是“无限备忘录”，而是一套有层次、有生命周期、有晋升和回收机制的记忆架构。

这个想法最开始可以用 JVM 的分代内存管理来类比。JVM 不会把所有对象都放在同一个地方，而是根据对象的生命周期，把它们放在不同区域。新对象先进入新生代，存活时间长、反复被使用的对象会晋升到老年代；一些特别大的对象可能会直接进入老年代；没用的对象则会被垃圾回收器清理掉。

Agent 记忆其实也可以类似。不是所有信息都应该直接进入长期记忆，也不是所有临时信息都应该马上丢掉。信息应该先进入短期区域，被观察一段时间，然后根据它的使用频率、重要程度和未来价值，决定它最终是被删除、归档、压缩、晋升，还是整理成 skill。

## 1. 当前上下文：只服务于眼前任务

Agent 的当前对话上下文，类似于运行时的临时工作区。它保存的是正在处理的问题、刚刚给出的指令、当前任务的中间状态和短期推理线索。

比如用户现在让我修改一段文章，或者排查一个终端报错，这些信息在当前任务中非常重要，但任务结束后，大部分内容并不需要进入长期记忆。它们的价值是即时的，不是长期的。

所以当前上下文不应该被误认为长期记忆。它更像短时注意力：负责当下的连续性，但不负责永久保存。

## 2. 每日记忆：先把现场接住

真正有价值的是“每日记忆”这一层。

很多信息在刚出现的时候，我们其实无法立刻判断它有没有长期价值。比如今天配置了一个 SSH 路径、解决了一个权限问题、尝试了一个 TTS 方案、改了一个 Hermes skill。它们当下看起来可能只是临时操作，但过几天可能会发现：这个问题反复出现，这个路径以后总会用到，这个流程其实应该写成 skill。

因此，Agent 需要一个类似 OpenClaw daily memory 的区域，用来保存当天发生的重要过程。它不需要像长期记忆一样高度精炼，也不需要像原始聊天记录一样完整无筛选。它的作用是保存“现场感”：今天做了什么、为什么这么做、遇到了什么坑、最后怎么解决。

这一步非常关键。因为如果没有每日记忆，Agent 会丢掉很多过程性上下文；但如果直接把这些东西塞进长期记忆，又会造成污染。

所以每日记忆可以理解为 Agent 记忆系统里的“新生代”：新发生的信息先进入这里，暂时存活，等待后续观察。

## 3. 项目记忆：介于短期和长期之间

除了每日记忆，还应该有项目级或主题级记忆。

有些信息不是全局长期事实，但在某个项目里非常重要。比如某个项目的目录结构、分支策略、运行命令、部署环境、老师要求、实验参数。这些内容未必应该写进全局 `MEMORY.md`，因为它们只对特定项目有用；但它们也不能只停留在某一天的 daily note 里，因为项目可能持续几周甚至几个月。

所以更合理的做法是建立 topic memory 或 project memory。它可以保存某个项目周期内持续有用的上下文，并且只在相关任务中加载。

这样可以避免一个常见问题：全局长期记忆越来越臃肿，里面混着各种项目的临时细节。项目记忆相当于给不同主题建立隔离区，让信息在正确的范围内发挥作用。

## 4. 长期记忆：只保存真正会影响未来行为的信息

长期记忆应该非常克制。

它不应该保存流水账，也不应该保存所有操作细节。长期记忆真正应该保存的是那些稳定、重要、未来会反复影响 Agent 行为的信息。

比如：

```text
用户长期偏好
用户的写作风格要求
用户常用技术栈
用户的重要项目背景
用户明确要求以后遵守的规则
稳定的环境配置
反复出现的工作习惯
高成本踩坑经验
```

这类信息的共同点是：以后如果 Agent 忘了，就会明显影响回答质量或执行效果。

所以长期记忆的价值不是容量，而是质量。它应该像被压缩过的经验，而不是未经整理的记录。

从 JVM 类比看，这一层就像老年代。能进入这里的信息，要么已经在短期记忆中存活了一段时间，证明它确实经常有用；要么一开始就非常重要，值得直接保留。

## 5. 记忆晋升不能只看频率，还要看意义

这里有一个很重要的问题：如果只根据“出现频率”决定是否进入长期记忆，会漏掉很多真正重要的事情。

人类并不是只记住高频事件。我们也会记住一些只发生过一次但意义重大的事情。比如一次重要失败、一次关键选择、一个强烈偏好、一个改变项目方向的决定。这些事情不需要反复出现，也应该被长期保存。

所以 Agent 的记忆晋升机制应该有两条路径。

第一条是普通晋升路径：

```text
短期记录
→ 多次出现
→ 多次被使用
→ 被压缩总结
→ 晋升为长期记忆
```

比如用户反复要求“英文内容下面附中文翻译”，那这就是一个长期偏好，应该进入 `USER.md`。

第二条是特殊晋升路径：

```text
只出现一次
→ 但意义重大
→ 直接进入长期记忆
```

比如用户明确说“以后都按这个结构写”，或者某次配置成功是整个系统后续运行的基础，那么它虽然只发生一次，也应该被保存。

这个地方用 JVM 类比就很准确：**有些对象不是因为活得久才进入老年代，而是因为它一开始就足够特殊，可能直接进入老年代。**

Agent 记忆也是一样。
高频有用的信息应该晋升；低频但高意义的信息也应该晋升。

## 6. Skills：重复执行的事情不该只是记忆，而该变成能力

还有一类信息更特殊：它不应该只是被“记住”，而应该被整理成 skill。

比如 Agent 多次完成这些任务：

```text
通过 SSH 控制远程 Windows
配置 QQ bot
写 Hermes skill
处理 TTS 流程
修复 Git 分支冲突
搭建某类项目环境
```

如果这些经验只是作为文字记忆存在，下次仍然需要重新推理一遍，那其实很浪费。更好的做法是把它们整理成可复用流程：什么时候触发、先检查什么、执行哪些命令、常见错误是什么、失败后怎么回滚。

这就是 skill 的意义。

事实记忆解决的是：

```text
我知道什么？
```

技能记忆解决的是：

```text
我会怎么做？
```

一个长期 Agent 真正变强，不只是因为它记住了更多用户信息，而是因为它把重复任务沉淀成了稳定方法。这里也可以借用 JVM 里的 JIT 思想：当某段逻辑被反复执行，系统就应该把它优化成更高效、更固定的执行路径。

对于 Agent 来说，反复出现的任务不应该永远停留在聊天历史里，而应该被“编译”成 skill。

## 7. 历史会话：冷数据不该总是加载，但必须能找回来

还有一层是历史会话数据库。

它保存的是过去发生过的完整记录，但这部分内容不应该每次都加载到上下文里。因为绝大多数历史细节在当前任务里并不需要。如果全塞进 prompt，只会浪费上下文窗口，还会干扰判断。

但它也不能被彻底丢掉。因为有时候用户会问：“上次我们怎么弄的？”“之前那个报错是什么？”“我上周让你写的结构还记得吗？”

这时候就需要 session search 或 semantic retrieval，从冷数据里把相关内容捞出来。

所以历史会话更像归档层。它不参与日常思考，但在需要追溯时提供证据。

## 8. Review / Dreaming：记忆系统真正的核心

如果只是分层存储，还不够。真正重要的是定期整理。

每日记忆会不断增长，项目记忆会逐渐过期，长期记忆也可能变得不准确。系统必须定期做 review，也可以叫 dreaming、consolidation 或 memory GC。

它要做几件事：

```text
删除过期信息
合并重复记忆
压缩冗长描述
把高频有用信息晋升到长期记忆
把项目流程整理成 skill
把已经不准确的记忆替换掉
把只适用于某个项目的信息限制在项目记忆中
```

这一步相当于 Agent 记忆系统里的垃圾回收和整理机制。

没有这一步，daily memory 会变成垃圾堆，long-term memory 会被污染，skills 也不会自然形成。很多 Agent 记忆系统效果不好，不是因为它们不会保存，而是因为它们缺少持续整理和晋升机制。

## 9. 一个更完整的 Agent 分代记忆模型

综合起来，一个比较理想的 Agent 记忆系统可以这样设计：

```text
当前上下文
= 当前任务的临时工作区

Daily Memory
= 每日详细记录，保存短期现场

Project / Topic Memory
= 项目级上下文，只在相关任务中加载

Long-term Memory
= 精炼后的长期事实、偏好、稳定约定

Significant One-time Memory
= 虽然低频，但意义重大，直接长期保存

Skills
= 反复执行的流程和操作方法

Session Archive
= 完整历史记录，需要时搜索

Review / Dreaming / GC
= 定期整理、压缩、晋升、遗忘

Skill Improvement / JIT
= 把重复任务优化成稳定 skill
```

如果一定要保留 JVM 类比，可以放在辅助理解层：

```text
当前上下文 ≈ Stack
Daily Memory ≈ Young Generation
Project Memory ≈ Survivor / Mid Generation
Long-term Memory ≈ Old Generation
Significant One-time Memory ≈ Large Object Directly Promoted
Skills ≈ Metaspace / Method Area
Session Archive ≈ Heap Dump / Logs
Review / Dreaming ≈ Garbage Collector
Skill Optimization ≈ JIT Compiler
```

但这个类比只是帮助理解，不是重点。重点是：**Agent 的记忆应该有生命周期管理。**

## 10. 结论：好的记忆不是多，而是会沉淀

我觉得理想的 Agent 长期记忆，不应该追求“什么都记住”。那样只会让系统越来越重，越来越混乱。真正好的记忆系统，应该像人一样有选择性。

普通信息先进入短期记录；
反复出现的信息逐渐晋升；
意义重大的信息直接保留；
项目相关的信息放在项目上下文里；
重复执行的流程变成 skill；
过期和无用的信息被压缩或遗忘；
完整历史留在归档里，需要时再检索。

这样 Agent 才不会只是一个保存聊天记录的工具，而会逐渐形成自己的经验结构。

它会知道用户是谁，也知道用户常做什么；
它能回忆过去发生过的事，也能区分哪些只是临时细节；
它能保存重要偏好，也能把重复任务变成稳定技能。

最终，一个真正长期可用的 Agent，不应该像硬盘一样机械保存一切，而应该像一个会整理经验的人：记住习惯，记住重要时刻，忘掉噪声，把反复做的事情变成能力。这样它才会越用越顺手，而不是越用越臃肿，喵。
