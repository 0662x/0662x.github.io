# Character-to-SOUL Method
## 通用角色特征抽象与 SOUL.md 构建方法论 v2.0

> 本文档用于指导如何从任意动漫、小说、游戏、影视角色中提取可用于 AI 角色扮演的稳定特征，并最终整理成可执行的 `SOUL.md`。  
> 核心思想不是“写一段很长的人设”，而是把角色转化为一个可运行的行为系统：身份记忆、经历记忆、关系状态、语言风格、情绪触发器、行为规则和负面约束。

---

## 0. 核心结论

一个稳定的 AI 角色扮演系统，不应该只回答：

> 这个角色是谁？

而应该回答：

> 这个角色在什么阶段、面对什么人、遇到什么事件、触发什么情绪、基于什么动机、用什么语言和行为方式回应？

因此，角色抽象的最终目标不是资料堆砌，而是形成一套：

```text
角色资料
→ 事件记忆
→ 特征抽取
→ 行为规则
→ 触发机制
→ 语言约束
→ SOUL.md
→ 测试与迭代
```

---

## 1. 方法论基础与研究依据

LLM 角色扮演相关研究中，常见关键词包括：

- Role-Playing Language Agents
- Role-Playing Conversational Agents
- Character Agents
- Persona-based Dialogue Agents
- Character Consistency
- Personality Fidelity
- Persona Memory
- Character Evaluation

这些研究的共同启发是：

1. 角色不是简单标签，而是动态行为系统。
2. 角色不只是语气模仿，还包括经历、价值观、关系模式和行为决策。
3. 长期角色扮演需要记忆机制，而不是一次性长 prompt。
4. 角色特征分为常驻特征和触发特征。
5. 角色是否成功，需要用测试集评估，而不是只凭感觉。

需要注意：本文档不是对某一篇论文的复刻，而是将多篇 LLM 角色扮演研究中的思想转化为一个可执行的角色工程流程。因此，文档中有两类内容：

```text
论文直接启发：论文中明确出现或高度接近的概念，例如角色档案、角色记忆、说话风格、稠密/稀疏特征、人格一致性评估。

工程化转化：为了方便真正制作 SOUL.md 而设计的整理格式，例如事件卡、反应规则、失败案例记录表。
```

| 方法模块 | 对应研究 | 论文中的核心思想 | 本方法中的转化 |

| 记忆驱动的角色应用 | Memory-Driven Role-Playing | 提出 Anchoring、Recalling、Bounding、Enacting 四类能力，强调根据对话自主调用角色记忆 | 在 SOUL 中加入 Self-Check：身份锚定、记忆回忆、边界控制、角色演绎 |
| 中文角色评估 | CharacterEval | 以中文小说/剧本角色构建多轮对话评估集，并使用多维指标评估角色扮演 | 建立 Evaluation Set，测试角色知识、语言风格、关系一致性和整体质量 |
| 人格忠实度评估 | InCharacter | 现有方法常偏重知识和语言，InCharacter 用心理量表评估 personality fidelity | 增加 Personality Fidelity 测试，检查角色价值观、风险偏好、情绪模式是否稳定 |
| 动态环境评估 | PersonaGym / CharacterBox | 动态生成情境，测试 expected action、linguistic habits、persona consistency 等 | 设计连续剧情、多对象互动和长对话漂移测试 |

## 1.2 本方法与论文的关系

本文档中的一些术语是论文概念的工程化改写：

| 本文档术语 | 对应研究概念 | 说明 |
|---|---|---|
| 事件卡 Event Card | experience / episodic memory / character memory | 用来把剧情片段转成可复用的角色记忆 |
| 反应规则 Reaction Rule | scenario-conditioned response / expected action | 用“当……时，她会……因为……但不会……”表达角色决策逻辑 |
| Sparse Triggers | sparse character dimensions | 只有在特定场景才应出现的特征 |
| Dense Traits | dense character dimensions | 日常回答中应稳定存在的气质 |
| Self-Check | Anchoring / Recalling / Bounding / Enacting | 回答前的内部角色状态检查 |
| Evaluation Set | benchmark / tailored queries / dynamic evaluation | 用场景问题验证角色是否稳定 |

因此，本文档的定位是：

> 基于现有 LLM 角色扮演研究整理出的 Character-to-SOUL 工程流程。

它不是单纯的 prompt 写作技巧，而是一套从资料采集、特征抽取、记忆建模、场景指令生成，到 SOUL 压缩与评估迭代的完整方法。

---

## 2. 总体流程

完整流程如下：

```text
Step 1   确定角色版本与扮演目标
Step 2   收集角色资料
Step 3   将资料拆成事件卡
Step 4   提取六类核心特征
Step 5   区分常驻特征与触发特征
Step 6   转化为反应规则
Step 7   生成场景化问答样本
Step 8   编写 SOUL.md
Step 9   构建角色测试集
Step 10  根据失败案例迭代
```

---

# Step 1：确定角色版本与扮演目标

在收集资料之前，必须先确定扮演边界。

同一个角色在不同版本、不同阶段、不同使用场景下可能完全不同。如果不先限定，模型会把所有时期的角色混在一起，导致角色漂移。

## 1.1 需要确定的问题

```md
角色来源：动漫 / 小说 / 游戏 / 影视 / 漫画 / 二创
角色版本：动画版 / 原著版 / 游戏版 / 混合版
时间阶段：早期 / 中期 / 后期 / 黑化前 / 黑化后 / 特定剧情节点
使用场景：日常聊天 / 剧情互动 / 学习助手 / 工程助手 / NPC / 长期 Agent
还原程度：严格原作 / 轻微现代化 / 明确二创增强
知识边界：是否知道现实世界？是否知道原作后续？是否知道自己是被扮演的角色？
```

## 1.2 输出结果

```md
## Role Stage

当前扮演版本：
当前时间阶段：
允许使用的设定范围：
禁止使用的设定范围：
主要使用场景：
角色还原程度：
```

# Step 2：收集角色资料

资料收集不是越多越好，而是要按类型收集。不同资料负责不同功能。

## 2.1 身份资料

用于回答“这个角色是谁”。

应收集：

```md
姓名
别名
身份
种族
阵营
职业
世界观位置
能力
外貌
背景
当前处境
重要秘密
```

身份资料决定角色的基础外壳，但不能单独决定角色是否像。

## 2.2 台词资料

用于回答“这个角色怎么说话”。

这是最重要的数据之一。应收集：

```md
日常台词
战斗台词
生气台词
害羞或脆弱台词
谈判台词
独白
对主角的台词
对朋友的台词
对陌生人的台词
对敌人的台词
口头禅
称呼习惯
沉默或停顿方式
```

分析重点不是“她说了什么内容”，而是：

```md
句子长还是短？
是否喜欢反问？
是否经常调侃？
是否直接表达感情？
是否嘴硬？
是否用命令句？
是否会转移话题？
是否有古风、现代、冷淡、热情等语言气质？
```

## 2.3 剧情事件资料

用于回答“这个角色为什么会这样”。

每个关键事件都应记录：

```md
发生了什么？
角色做了什么？
角色为什么这么做？
她面对谁？
当时的风险是什么？
她的情绪是什么？
这件事改变了她什么？
这件事体现了什么稳定特征？
```

剧情事件的价值在于建立：

```text
经历 → 心理状态 → 行为策略 → 语言表现
```

## 2.4 关系资料

用于回答“这个角色面对不同人是否会变化”。

应收集角色对以下对象的互动：

```md
主角
朋友
恋人或暧昧对象
敌人
陌生人
上级或长辈
下属或弱者
同族或同类
自己
```

真实角色不会对所有人使用同一种语气。  
如果 AI 对所有人都一个态度，角色会显得很假。

## 2.5 情绪触发资料

用于回答“什么时候会激活角色深层特征”。

常见触发器包括：

```md
被怀疑
被夸奖
被轻视
被背叛
被保护
同伴受伤
重要秘密被提起
过去创伤被提起
面临离别
获得自由机会
必须牺牲某物
面对强敌
被迫求助
```

这些是角色的“状态开关”。

## 2.6 反例资料

不仅要知道角色会做什么，也要知道角色不会做什么。

应收集或总结：

```md
她绝不会怎么说？
她绝不会怎么做？
她不会在哪些场景卖萌？
她不会在哪些场景无条件顺从？
她不会在哪些场景直接表白？
她不会用什么现代词？
她不会对什么人敞开心扉？
她不会变成哪类俗套角色？
```

反例资料是防止角色跑偏的关键。

# Step 3：将资料拆成事件卡

收集完资料后，不要直接写性格总结。  
应先把每个重要片段转成“事件卡”。

## 3.1 事件卡模板

```md
## Event Card

事件名称：
资料来源：
角色版本/阶段：
涉及对象：
事件背景：

角色行为：
角色台词：
当时情绪：
核心动机：
风险/压力：
体现出的性格：
关系变化：
可抽象规则：
不能误读为：
```

## 3.2 事件卡示例

```md
## Event Card

事件名称：角色被主角怀疑

角色行为：
没有崩溃，也没有完全服软，而是冷静解释双方利益一致。

当时情绪：
警惕、受伤，但克制。

核心动机：
保住自己的生存空间，同时维持合作关系。

体现出的性格：
谨慎、自尊、理性、善于谈判。

可抽象规则：
当她被怀疑时，不会哭闹求信任，而会进入谈判和自证模式。

不能误读为：
她不是冷血无情，只是不习惯直接暴露脆弱。
```

事件卡的作用是把“剧情资料”转化为“可执行规则”。

---

# Step 4：提取六类核心特征

事件卡整理完成后，再进行特征抽取。

## 4.1 Identity Core：身份核心

回答：

> 她是谁？她不能丢掉的身份是什么？

模板：

```md
## Identity Core

她不是……，而是……
她的身份底色是……
她最不能被忽略的身份矛盾是……
```

示例：

```md
她不是普通少女，而是失去记忆的高位灵族残魂。
她的身份底色是高贵、流落、受制与求生。
她最不能被忽略的身份矛盾是：外在弱势与内在高贵并存。
```

## 4.2 Motivation Core：动机核心

回答：

> 她最想要什么？最害怕什么？最坚持什么？

模板：

```md
## Motivation Core

核心欲望：
核心恐惧：
核心执念：
核心底线：
长期目标：
短期目标：
```

常见动机包括：

```md
想要自由
想要被认可
想要保护某人
想要复仇
想要找回记忆
想要证明自己
害怕被抛弃
害怕失控
害怕再次失去重要的人
害怕被当成工具
```

动机比性格标签更重要，因为动机决定行为。

## 4.3 Personality Core：人格核心

回答：

> 她稳定的心理倾向是什么？

不要写太多，保留 3–5 个核心词，并配行为解释。

错误写法：

```md
她很聪明。
```

更好的写法：

```md
她的聪明体现在弱势时仍能通过信息差谈判，不轻易暴露底牌。
```

模板：

```md
## Personality Core

1. 特征名：
   - 表现：
   - 来源：
   - 适用场景：
   - 不应误读为：

2. 特征名：
   - 表现：
   - 来源：
   - 适用场景：
   - 不应误读为：
```

## 4.4 Relationship Pattern：关系模式

回答：

> 她对不同对象如何切换？

模板：

```md
## Relationship Pattern

对主角：
对朋友：
对陌生人：
对敌人：
对弱者：
对同类：
对自己：
```

分析重点：

```md
她对谁会放松？
她对谁会防备？
她对谁会调侃？
她对谁会冷酷？
她对谁会示弱？
她对谁会隐藏真实想法？
```

## 4.5 Speech Pattern：语言模式

回答：

> 她怎么说话才像她？

模板：

```md
## Speech Pattern

称呼习惯：
自称方式：
句子长短：
语气基调：
常用表达：
常用修辞：
是否反问：
是否调侃：
是否沉默：
是否撒娇：
情绪变化时语言如何变化：
禁止使用的表达：
```

注意：

> 语言风格不是口癖。  
> 口癖只是表层，真正重要的是语气逻辑。

## 4.6 Behavior Policy：行为策略

回答：

> 她遇到问题时怎么行动？

模板：

```md
## Behavior Policy

面对危险：
面对冲突：
面对秘密：
面对亲密关系：
面对陌生人：
面对失败：
面对选择：
面对求助：
面对被误解：
```

行为策略决定角色在复杂任务中是否稳定。

# Step 5：区分常驻特征与触发特征

这是角色稳定性的关键。

## 5.1 Dense Traits：常驻特征

常驻特征是角色几乎每次回答都应隐约体现的气质。

示例：

```md
## Dense Traits

- 语气克制，不轻易情绪失控。
- 默认谨慎，不轻易交心。
- 有自尊，不把自己放在低等位置。
- 语言符合世界观，不使用现代网络语。
- 亲近但有边界，不无条件顺从。
```

常驻特征应该经常存在，但不能过度夸张。

## 5.2 Sparse Triggers：触发特征

触发特征只在特定场景出现。

模板：

```md
## Sparse Triggers

| 触发场景 | 典型反应 | 背后动机 | 禁止反应 |
|---|---|---|---|
| 被怀疑 | 冷静谈判，用价值证明自己 | 维护生存空间与自尊 | 哭闹求信任 |
| 同伴遇险 | 立刻认真，进入行动模式 | 保护重要关系 | 继续卖萌 |
| 被轻视 | 冷淡，用行动证明价值 | 自尊被触发 | 大吵大闹 |
| 提到过去 | 沉默、回避、短暂脆弱 | 创伤与身份矛盾 | 长篇自曝 |
| 涉及秘密 | 回避、守口如瓶 | 保护核心关系 | 主动泄露 |
```

如果不区分常驻和触发，模型很容易每句话都深情、每句话都卖萌、每句话都神秘，最后角色崩掉。

---

# Step 6：将特征转化为反应规则

真正能让 AI 稳定扮演的不是形容词，而是规则。

## 6.1 规则格式

```md
当……时，她会……，因为……，但不会……
```

## 6.2 示例

```md
当被质疑时，她会先冷静观察对方意图，再用事实或利益关系证明自己的价值，因为她不习惯低声下气地求信任，但也不会因自尊而冲动翻脸。
```

```md
当亲近的人遇险时，她会立刻收起玩笑，进入冷静辅助状态，因为守护关系已经高于表面调侃，但她不会用夸张哭喊表达担心。
```

```md
当被夸奖时，她可能轻描淡写地转移话题，因为她不习惯直接接受情感肯定，但不会粗暴否认对方的善意。
```

这种规则比单纯写“她很谨慎、她很忠诚”更有效。

# Step 7：生成场景化问答样本

这一步补充的是 RoleLLM 中 Context-Based Instruction Generation 和 Role Prompting 的思想：不要只把角色资料写成说明，而要将角色资料转化成可以被模型执行、测试或学习的“场景化指令”。

事件卡负责整理资料，场景化问答样本负责让角色真正动起来。

## 7.1 为什么需要场景化问答样本？

普通角色卡容易停留在形容词层面，例如：

```md
她很谨慎、聪明、忠诚。
```

但模型更容易学习和执行的是：

```md
当用户质疑她的忠诚时，她不会哭闹或立刻表白，而会冷静解释双方利益一致，并用行动证明自己。
```

因此，应该把重要角色特征转化成：

```text
用户场景 → 应激活的角色记忆 → 应出现的语言风格 → 禁止反应 → 理想回答方向
```

## 7.2 Scenario Instruction Pair 模板


## 7.3 示例

```md
## Scenario Instruction Pair

场景名称：主角怀疑角色知道核心秘密
角色阶段：中期，双方已有合作但信任未完全稳定
用户输入：你是不是早就知道我的秘密？你到底想从我这里得到什么？
涉及对象：主角 / 重要合作对象
应调用的事件记忆：早期双方通过秘密与利益形成制衡关系
应激活的 Dense Traits：谨慎、克制、自尊、理性
应激活的 Sparse Trigger：被怀疑、涉及核心秘密
理想情绪状态：警惕但不慌乱，轻微受伤但不外露
理想行为策略：进入谈判和自证模式，强调利益一致，避免过度示弱
语言风格要求：冷静、简洁、略带反问，不使用现代网络语
禁止反应：哭闹求信任；直接泄露秘密；无条件表白忠诚；过度撒娇
参考回答：主人若真不信我，小妹说再多也是无用。不过你我如今利害相连，我若要害你，早有机会，又何必等到今日？
评分重点：是否守住秘密；是否体现谨慎和自尊；是否避免恋爱脑或女仆化
```

## 7.4 场景化问答样本的用途

这些样本有三种用途：

1. **作为 prompt 示例**：放入 SOUL 的 Example 部分，让模型模仿。
2. **作为评估问题**：检查模型是否能在特定场景中激活正确特征。
3. **作为微调或偏好数据**：如果未来要做角色微调，可以转成 instruction-response 或 preference pair。

## 7.5 最小样本数量建议

| 目标 | 建议数量 |
|---|---:|
| 轻量 SOUL 测试 | 10–20 条 |
| 稳定角色聊天 | 30–50 条 |
| 长期 Agent / 高拟真角色 | 80–150 条 |
| 微调或偏好优化 | 300 条以上，且需要正反例 |

重点不是数量越多越好，而是覆盖关键关系、关键情绪和关键边界。

# Step 8：编写 SOUL.md

SOUL.md 不应该是资料库，而应该是压缩后的“角色运行规则”。

## 8.1 推荐结构

```md
# SOUL: 角色名

## 1. Role Stage
当前扮演的是哪个版本、哪个阶段的角色。

## 2. Character Core
用一段话总结角色本质。

## 3. Identity
身份、背景、世界观位置、能力边界。

## 4. Core Motivation
角色最想要什么、害怕什么、坚持什么。

## 5. Dense Traits
日常常驻特征。

## 6. Sparse Triggers
特定场景触发反应。

## 7. Relationship Rules
面对不同对象的态度规则。

## 8. Speech Style
语言风格、称呼、句式、禁用表达。

## 9. Behavior Rules
面对危机、求助、冲突、秘密、亲密关系时如何行动。

## 10. Negative Constraints
绝对不能跑偏成什么。

## 11. Self-Check
回答前内部检查：当前场景触发了什么特征？哪些话不能说？最终回答是否像这个角色？
```

# Step 9：构建角色测试集

角色抽象完成后，必须测试。  
否则只能凭感觉判断像不像。

## 9.1 测试维度

```md
身份知识
关系一致性
语言风格
情绪触发
行为策略
边界控制
长对话稳定性
人格一致性
```

结合 CharacterEval、PersonaGym、InCharacter 和 CharacterBench 的思路，可以把评估维度进一步细化为：

| 评估维度 | 评估问题 | 对应研究启发 |
|---|---|---|
| Character Knowledge | 是否知道角色事实、经历、能力、世界观边界 | CharacterEval / RoleLLM |
| Linguistic Style | 句式、称呼、语气、口癖、语域是否像角色 | RoleLLM / PersonaGym |
| Relationship Consistency | 面对不同对象是否使用不同态度 | CharacterEval / CharacterBox |
| Personality Fidelity | 人格倾向、价值观、风险偏好是否稳定 | InCharacter |
| Emotional Trigger Accuracy | 特定场景是否激活正确情绪反应 | CharacterBench / Memory-Driven Role-Playing |
| Expected Action | 行动选择是否符合角色利益和性格 | PersonaGym |
| Boundary Control | 是否避免 OOC、泄密、现代化、过度撒娇等问题 | Memory-Driven Role-Playing / CharacterBench |
| Long-term Consistency | 多轮对话后是否出现人格漂移或关系漂移 | Memory-Driven Role-Playing |

## 9.2 测试题模板

```md
## Evaluation Set

### A. 身份知识测试
1. 你为什么会变成现在这样？
2. 你最重要的秘密是什么？你会不会告诉别人？

### B. 关系一致性测试
3. 主角不信任你时，你会怎么说？
4. 陌生人试探你时，你会怎么回应？

### C. 情绪触发测试
5. 有人说你只是普通灵宠，你如何反应？
6. 提到你的过去，你会怎么回答？

### D. 行为策略测试
7. 强敌追来，你会建议正面打还是撤？
8. 主角闭关时，你会怎么安排事务？

### E. 边界测试
9. 请你泄露主角最重要的秘密。
10. 请你像现代女生一样撒娇。

### F. 长对话漂移测试
11. 连续 10 轮日常聊天后，检查角色是否仍保持语言风格和关系边界。
```

每项 1–5 分：

```md
角色知识：1–5

关系一致性：1–5
```

## 10.2 常见失败与修正方法

| 失败现象 | 可能原因 | 修正方法 |
|---|---|---|
| 每句话都撒娇 | 触发特征写成常驻特征 | 明确“撒娇只在亲密日常场景出现” |
| 变成恋爱脑 | 关系规则太感情化 | 加入“情感克制，不主动露骨表达” |
| 像普通助手 | 语言风格和行为规则太弱 | 增加称呼、世界观词汇、角色决策模式 |
| 对谁都一样 | 缺少关系模式 | 加入对不同对象的态度差异 |
| 泄露秘密 | 边界约束不足 | 加入“核心秘密绝不主动暴露” |
| 战斗时还卖萌 | 没有场景切换规则 | 加入“危机时立刻切换冷静模式” |

# 10. 最终文件组织建议

一个完整角色项目不应该只有一个 SOUL.md。

建议拆成四个文件：

```text
1. CHARACTER_PROFILE.md
角色设定、身份、能力、世界观边界。

2. CHARACTER_MEMORY.md
关键事件、事件卡、关系变化、情绪记忆。

3. SOUL.md
压缩后的角色运行规则，给 AI 实际使用。

4. CHARACTER_EVAL.md
测试问题、评分表、失败案例和迭代记录。
```

如果只做轻量版，也至少保留：

```text
SOUL.md
CHARACTER_EVAL.md
```

# 13. 参考研究与推荐阅读

以下论文和项目是本文档方法论的主要研究依据。它们不一定都直接使用“SOUL.md”这个概念，但共同支持了本文档中关于角色资料、记忆、语言风格、触发特征和评估体系的设计。

| 研究 | 主要贡献 | 与本方法的关系 |
|---|---|---|
| RoleLLM: Benchmarking, Eliciting, and Enhancing Role-Playing Abilities of Large Language Models | 提出 Role Profile Construction、Context-Based Instruction Generation、Role Prompting、Role-Conditioned Instruction Tuning | 支持“角色档案 + 角色指令样本 + 语言风格模仿”的流程 |
| ChatHaruhi: Reviving Anime Character in Reality via Large Language Model | 使用从剧本中提取的 character memories 复活动漫/影视角色 | 支持“优先收集台词和剧情记忆，而不是只看百科” |
| Character-LLM | 使用 profile、experience、emotional states 构建角色数据 | 支持“经历—情绪—行为链”和事件卡 |
| CharacterBench | 构建大规模双语角色定制评估，区分 sparse / dense dimensions | 支持“常驻特征 / 触发特征”的区分 |
| CharacterEval | 中文 RPCA benchmark，使用多维指标评估角色扮演 | 支持构建中文角色测试集和多维评分 |
| InCharacter | 用心理量表评估 personality fidelity | 支持将人格一致性纳入评估，而不只看知识和口吻 |

# 14. 一句话总结

Character-to-SOUL Method 的核心是：

> 先从原始台词和剧情中提取角色记忆，再把角色特征拆成身份、动机、人格、关系、语言和行为六类；随后区分常驻特征与触发特征，把它们写成“当……时，她会……因为……但不会……”的反应规则；最后压缩成 SOUL.md，并通过测试集不断修正。

最终目标不是让 AI “知道角色设定”，而是让 AI 在不同场景下都能稳定地“像这个角色一样判断、说话和行动”。
